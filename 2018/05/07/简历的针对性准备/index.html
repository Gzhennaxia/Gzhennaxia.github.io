面向对象设计原则

面向对象设计原则（五大原则SOLID）

  1. Single Responsibility Principle——单一职责原则

    核心思想: 一个类应该只有一个引起它变化的原因。

    一个类只负责一个功能领域中的相应职责

    自己负责领域的事情都做，不是自己领域的事不去插手。

    单一职责原则是实现高内聚、低耦合的指导方针。

  2. Open Closed Principle——开放封闭原则

    核心思想:对扩展开放,对修改封闭.

    抽象化是开闭原则的关键。

  3. Liskov Subsitution Principle——里氏替换原则

    核心思想: 子类必须能够替换掉它们的父类.

  4. InterFace Segregation Principle——接口隔离原则

    核心思想:使用多个小的专门的接口,而不要使用一个大的总接口。

  5. Dependency Inversion Principle——依赖倒置原则

    核心思想:抽象不应依赖于细节，细节应该依赖于抽象。

DI、IoC、AOP

依赖注入DI（Dependecy Injection）和控制反转IoC（Inversion of Control）是同一个概念。

具体的讲：当某个角色需要另外一个角色协助的时候，在传统的程序设计过程中，通常由调用者来创建被调用者的实例。但在spring中创建被调用者的工作不再由调用者来完成，因此称为控制反转。创建被调用者的工作由spring来完成，然后注入调用者，因此也称为依赖注入。

依赖注入的四种方式

  1. set方法注入
  2. 构造函数注入
  3. 静态工厂注入（调用静态工厂的静态方法创建bean）
  4. 实例工厂注入（先实例化实例工厂，然后调用实例工厂的普通方法创建bean）

之所以有工厂注入方式是因为：spring并不是JavaEE规范，但为了在重构时使用spring，所以提供了工厂注入方式

AOP的三种实现方式

  1. 手动方式(自己实现)

    JDK动态代理（基于接口）

    CGLib动态代理（基于继承）

  2. 半自动方式(SpringAOP了解)

    SpringAOP既支持JDK动态代理也支持CGLib动态代理

    注意：在配置interceptorNames时要用value来赋值，而不是ref因为在ProxyFactoryBean中interceptorNames是一个字符串数组（基本数据类型）

  3. 自动方式（AspectJ掌握）

JVM内存模型

http://gityuan.com/2016/01/09/java-memory/

Java虚拟机(Java Virtual Machine=JVM)的内存空间分为五个部分，分别是：

  1. 程序计数器
  2. Java虚拟机栈
  3. 本地方法栈
  4. 堆
  5. 方法区

  1. 程序计数器的特点

    1. 是一块较小的存储空间
    2. 线程私有。每条线程都有一个程序计数器。
    3. 是唯一一个不会出现OutOfMemoryError的内存区域。
    4. 生命周期随着线程的创建而创建，随着线程的结束而死亡。

  2. Java虚拟机栈

    1. 当一个方法即将被运行时，Java虚拟机栈首先会在Java虚拟机栈中为该方法创建一块“栈帧”，栈帧中包含局部变量表、操作数栈、动态链接、方法出口信息等
    2. Java虚拟机栈是由一个个栈帧组成
    3. Java虚拟机栈会出现两种异常：StackOverFlowError和OutOfMemoryError。
      a) StackOverFlowError：
      若Java虚拟机栈的内存大小不允许动态扩展，那么当线程请求栈的深度超过当前Java虚拟机栈的最大深度的时候，就抛出StackOverFlowError异常。
      b) OutOfMemoryError：
      若Java虚拟机栈的内存大小允许动态扩展，且当线程请求栈时内存用完了，无法再动态扩展了，此时抛出OutOfMemoryError异常

  3. 本地方法栈

    本地方法栈和Java虚拟机栈实现的功能类似，只不过本地方法区是本地方法运行的内存模型。

  4. 堆

    1. 堆是用来存放对象的内存空间
    2. 堆的特点
      1. 线程共享
        整个Java虚拟机只有一个堆，所有的线程都访问同一个堆
      2. 在虚拟机启动时创建
      3. 垃圾回收的主要场所。
      4. 可以进一步细分为：新生代、老年代。
        新生代又可被分为：Eden、From Survior、To Survior。
        不同的区域存放具有不同生命周期的对象。这样可以根据不同的区域使用不同的垃圾回收算法，从而更具有针对性，从而更高效。
      5. 堆的大小既可以固定也可以扩展，但主流的虚拟机堆的大小是可扩展的，因此当线程请求分配内存，但堆已满，且内存已满无法再扩展时，就抛出OutOfMemoryError。

  5. 方法区

    1. Java虚拟机规范中定义方法区是堆的一个逻辑部分

    2. 方法区中存放三种数据：类信息、常量、静态变量

    3. 方法区特点

      1. 线程共享
        方法区是堆的一个逻辑部分，因此和堆一样，都是线程共享的。整个虚拟机中只有一个方法区。

      2. 永久代
        方法区中的信息一般需要长期存在，而且它又是堆的逻辑分区，因此用堆的划分方法，我们把方法区称为老年代。

      3. 内存回收效率低
        方法区中的信息一般需要长期存在，回收一遍内存之后可能只有少量信息无效。

        对方法区的内存回收的主要目标是：对常量池的回收 和 对类型的卸载。

      4. Java虚拟机规范对方法区的要求比较宽松。
        和堆一样，允许固定大小，也允许可扩展的大小，还允许不实现垃圾回收。

垃圾回收机制

  1. 判断一个对象是否可回收
    1. 引用计数
    2. 可达性
  2. 垃圾回收算法
    1. 标记-清除法
    2. 标记-整理法
    3. 复制
  3. 分代收集
    1. 新生代使用：复制算法
    2. 老年代使用：标记 - 清理 或者 标记 - 整理 算法

MD5算法

MD5即Message-Digest Algorithm 5（信息-摘要算法5）

MD5算法是单向散列算法的一种。单向散列算法也称为HASH算法，是一种将任意长度的信息压缩至某一固定长度（称之为消息摘要）的函数(该压缩过程不可逆)

抢红包算法

  1. 设置红包最值

  2. 随机产生红包

    1. 保证红包总和固定

      线段切割法

    2. 保证每次随机金额的平均值相等

      二倍均值法（除了最后一个人以外其他人强盗的金额都不超过人均金额的两倍）

每次抢到的金额=随机区间（0， 剩余金额）

推荐算法

目的：

  1. 发掘长尾（帮助用户找到想要的东西）
  2. 降低信息过载
  3. 提高点击率
  4. 加深对用户的了解

分类：

  1. 基于流行度的算法
  2. 协同过滤算法(Collaborative Filtering, CF )
    1. 通过找到兴趣相投或者有共同经验的群体想用户推荐感兴趣的信息
    2. 步骤：
      1. 找到用户A的兴趣爱好集合
      2. 找到与用户A有相同兴趣爱好的用户群体
      3. 找到该群体的兴趣爱好集合
      4. 将这个集合推荐给用户A
  3. 关联推荐（啤酒与尿布）
    1. 通过数据挖掘，发现数据之间的关联
    2. 目的：用户购买A后推荐B，比单独推荐B，过去更好的效果。
    3. 步骤：
      1. 计算关联规则的支持度（同时购买这两种商品的概率）
      2. 计算关联规则的置信度（条件概率，已知购买了A，有多大概率购买了B）
      3. 计算关联规则的提升度（A–>B关联规则推荐，与直接推荐B的比值，即推荐效果）
  4. 基于内容的算法
  5. 基于模型的算法
  6. 混合算法

Spring Boot 框架介绍和使用

Spring Boot Reference Guide

https://blog.csdn.net/xlgen157387/article/details/52830071

简单的介绍：

Spring Boot是在Spring框架基础上创建的一个全新的框架，其设计目的是简化Spring应用的搭建和开发过程。它的特点是遵循“习惯/约定优于配置”的原则，使用少量的配置就可以迅速构建一个企业级应用程序。

目的：简化新 Spring 应用的初始搭建以及开发过程

​ 核心：自动配置

spring boot的核心功能：

  1. 独立运行的spring项目

    Spring Boot可以以jar包的形式进行独立的运行

  2. 内嵌servlet容器

    内嵌容器，使得我们直接执行项目的主方法main函数，就可以实现项目的快速运行；

  3. 提供starter简化Manen配置

    Spring Boot提供了一系列的starter pom用来简化我们的Maven依赖

  1. 自动配置Spring

    Spring Boot会根据我们项目中类路径的jar包/类，为jar包的类进行自动配置Bean，这样一来就大大的简化了我们的配置。

  2. 无代码生成和XML配置

    Spring Boot神奇的地方不是借助于代码生成来实现的，而是通过条件注解的方式来实现的。

（1）遵循“习惯优于配置”的原则，使用Spring Boot只需要很少的配置，大部分的时候我们直接使用默认的配置即可；

（2）项目快速搭建，可以无需配置的自动整合第三方的框架；

（3）可以完全不使用XML配置文件，只需要自动配置和Java Config；

（4）内嵌Servlet容器，降低了对环境的要求，可以使用命令直接执行项目，应用可用jar包执行：java -jar； （5）提供了starter POM, 能够非常方便的进行包管理, 很大程度上减少了jar hell或者dependency hell；

（6）运行中应用状态的监控；

（7）对主流开发框架的无配置集成；

（8）与云计算的天然继承；
